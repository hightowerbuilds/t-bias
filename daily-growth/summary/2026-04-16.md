### 4. Made New Tabs Inherit the Current Working Context
Changed tab creation so new tabs open in the same path as the currently focused workspace when possible:

- Terminal tabs inherit the live shell CWD
- File explorer tabs inherit their current root path
- Editor tabs inherit the parent directory of the open file

This keeps tab creation local to the user’s current task instead of bouncing back to a default home directory.

### 5. Added Foreground Process Detection for Terminal Tab Titles
Implemented backend and frontend wiring so terminal tabs can temporarily adopt the name of the foreground app running inside the PTY:

- Detects apps like `claude`, `codex`, `vim`, `nvim`, `less`, `tmux`, and similar tools
- Suppresses plain shell names so the tab only changes when a meaningful foreground app takes over
- Falls back to the underlying pane title when the shell regains control

This makes tab titles more informative during agent/tool-heavy sessions.

### 6. Built Prompt Stacker
Introduced a new prompt-capture feature for reusable prompts:

- Added prompt input and saved-prompt list UI
- Added Rust-backed persistence so prompts are written to the app’s data directory on disk
- Added a shared Solid store for prompt state instead of keeping state trapped in a single component instance
- Fixed focus behavior so paste and dictation target the textarea when Prompt Stacker opens

The persistence path is platform app data, which is the correct long-term location for shipped app-owned state on macOS rather than writing into the Desktop.

### 7. Added a Session Landing Surface and Shell Switching
Reworked session flow so users can leave the current workspace and return to a landing screen without closing the app:

- Added a dedicated landing surface for resuming the current auto-saved workspace
- Added direct loading for named sessions
- Added a clean `New Session` path
- Exposed the launcher in the top bar, later renamed from `Sessions` to `Shells`

This turned session restore from a startup-only prompt into an actual in-app navigation surface.

### 8. Moved Prompt Stacker from a Tab into a Modal Workspace
After the first Prompt Stacker version shipped as a tab, refactored it into a centered overlay that behaves more like the session landing surface:

- Removed tab-first behavior for opening Prompt Stacker
- Reused the shared prompt store and persistence layer
- Added modal presentation, focus-on-open, click-outside close, `Escape` close, and `Done` dismissal
- Updated older saved Prompt Stacker tabs to restore as shell tabs so the tab-based version does not keep resurfacing

The launcher was also renamed from a right-arrow glyph to a labeled `Stacker` button.

## Key Decisions

**Why Prompt Stacker is persisted in app data instead of the Desktop:**  
The Desktop is user-facing content space, not app-owned state. Using the platform config/app-data directory keeps prompt storage stable, private to the app, and aligned with how sessions/config are already stored.

**Why Prompt Stacker moved from a tab to a modal:**  
It reads more like a supporting workspace tool than a peer to shell/editor/file tabs. Making it modal keeps the tab bar focused on core work surfaces while still keeping Prompt Stacker close at hand.

**Why markdown preview uses the app background:**  
The custom gradient looked attractive, but it visually detached the preview from the rest of t-bias. Using the shared theme background keeps the reading mode integrated with the app’s overall visual language.
